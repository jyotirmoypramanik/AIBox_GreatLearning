🤖 OpenAI Client — the "AI phone line"

Created once, reused by all three stages. Points to a course API endpoint.

client = OpenAI(
  base_url="https://openai.vocareum.com/v1",
  api_key="your-key-here"
)
MODEL = "gpt-4.1-nano"   # model used throughout

⚙️ get_completion() — the shared AI helper every stage calls

All three stage functions call this helper. It builds the message list, fires the API request, and returns the AI's raw text response.

def get_completion(
    messages    = None,       # 👈 list of role/content dicts
    system_prompt = None,     # optional — prepended as "system"
    user_prompt   = None,     # optional — appended as "user"
    model         = MODEL     # default: gpt-4.1-nano
) -> str:

    response = client.chat.completions.create(
        model=model,
        messages=messages,
        temperature=0.7
    )
    return response.choices[0].message.content  # 👈 raw text back
What the message list looks like — every API call has at least two messages:
{"role":"system", "content": "You are an insurance assistant…"} ← tells AI its job
{"role":"user", "content": "Claim ID: C001 … windshield…"} ← the actual data

🔗 Why chain prompts?

One giant prompt gets confused. Three small specialised prompts — each with a single clear job — are far more reliable and easier to debug.

📐 What is Pydantic?

A Python library that acts like a strict form. You define exactly what fields are allowed, their types and constraints. Any AI response that doesn't match is rejected automatically.

🧠 What is Prompt Engineering?

Carefully writing the system message so the AI knows its role, the rules it must follow, and the exact format (JSON) it must return.

🛡️ What are Gate Checks?

Python code that runs after every AI call to verify the output before passing it to the next stage. Stops bad data from silently propagating.

What Pydantic is doing here: It acts like a customs officer. The AI's raw JSON text is first parsed (json.loads) then fed into ClaimInformation(**dict). Pydantic then checks every field — correct type? within length? damage_area value in the allowed list? If anything fails, it raises an error and the gate catches it, returning None instead of passing bad data forward.
What Prompt Engineering is doing here: The system message shapes the AI's entire behaviour for this call — it defines the AI's role, the output format (JSON only), and the exact allowed values for damage_area. Without this, the AI might say "the windshield" instead of "windshield", which would fail Pydantic validation.

Then all three lists zipped into a pandas DataFrame for the final display.
What Pydantic is doing here: The queue field uses Literal["glass","fast_track","material_damage","total_loss"] — it acts like a whitelist. If the AI returns anything else (e.g., "fast track" with a space), Pydantic rejects it immediately. The priority field uses ge=1, le=5 (greater-than-or-equal / less-than-or-equal) constraints from Field(), so 0 or 6 are automatically rejected.
What Prompt Engineering is doing here: The system prompt gives the AI both the routing rules AND the priority scale, so the AI is reasoning with the carrier's business logic baked in. Crucially, the user message contains the full combined object (claim + severity) as JSON — giving the AI everything it needs to make an informed routing decision without any prior context.

🎯 What the whole notebook achieves — in one sentence

Five raw, informal accident reports go in → three chained AI calls (each with a specialist prompt + a Pydantic gate) transform them into fully validated, structured routing decisions → a pandas DataFrame gives the insurance operations team an instant, actionable triage list — all without a single human reading the reports manually.
