Stage 1 — Information Extraction
AI reads the messy free-text report and pulls out the key facts

The AI is given a system prompt that says: "You are an insurance assistant. Read this accident report and return a structured JSON object." It converts each informal paragraph into clean, machine-readable fields.

📩 What goes in
  • Raw text accident report
  • Could be typed by anyone — messy, informal language
🤖 What the AI does
  • Finds the claim_id
  • Extracts customer name & vehicle
  • Summarises loss_desc
  • Tags damage_area (windshield, bumper, hood…)
📤 What comes out
  • Structured JSON, e.g.:
    damage_area: ["windshield"]
  • One clean object per claim
🛡️
Gate 1 — Schema Validation
Pydantic checks: claim_id 2–10 chars? damage_area uses only allowed values? If wrong → reject & log error, don't continue.
▼
📊
Stage 2 — Severity Assessment
AI estimates how bad the damage is and what it will likely cost

The structured claim from Stage 1 is handed to the AI again with a new prompt: "You are a damage assessor. Given these damage details, classify the severity and estimate repair cost." The AI applies the carrier's own heuristics.

📩 What goes in
  • Validated claim object from Gate 1
  • Damage description & areas
🤖 What the AI does
  • Minor → scratches, chips ($100–$1,000)
  • Moderate → bumper, panels ($1,000–$5,000)
  • Major → structural, total loss ($5,000–$50,000)
📤 What comes out
  • severity: "Minor"
  • est_cost: 250.00
  • One assessment per claim
🛡️
Gate 2 — Cost-Range Check
Code verifies: does the estimated cost actually fall in the right bracket for the stated severity? E.g. "Minor" but cost = $8,000 → rejected.
▼
🗂️
Stage 3 — Queue Routing
AI decides where to send the claim and how urgent it is

The claim + severity data are passed to a third AI call with a final prompt: "Route this claim to the correct processing queue and assign a priority 1–5." Each queue is handled by different specialist teams inside the insurance company.

📩 What goes in
  • Claim info + severity + cost
🤖 Routing rules
  • 🔵 glass — Minor, only glass damage
  • 🟢 fast_track — Other minor damage
  • 🟣 material_damage — All moderate
  • 🔴 total_loss — All major
📤 What comes out
  • queue: "glass"
  • priority: 4 (1=urgent, 5=low)
🛡️
Gate 3 — Routing Validation
Pydantic checks queue is one of the 4 allowed values; priority is integer 1–5. Rejects anything that doesn't conform.

💡 Key Concepts Used in This Notebook
🔗

Prompt Chaining

Instead of one big complicated prompt, the problem is split into 3 smaller, specialised AI calls — each one focused on a single job. The output of each step feeds the next.

🛡️

Gate Checks

After every AI step, a code-based validator (using Pydantic) checks that the AI's answer makes sense. If it doesn't pass, the chain stops — preventing bad data cascading downstream.

📐

Pydantic Models

Think of these as strict forms with rules — e.g. "severity must be Minor, Moderate, or Major — nothing else". They enforce structure on the AI's free-text JSON output.

🤖

Agentic Reasoning

The system reasons step-by-step like a human expert would: first understand the claim, then judge how bad it is, then decide where to send it — each step informing the next.

🎯 What Is This Notebook Achieving?

It automates the first stage of insurance claims processing — a task that traditionally requires a human agent to manually read each report, classify the damage, and route it to the right team.

By chaining three AI prompts together (extract → assess → route), with automated gate checks in between, the system can process hundreds of messy accident reports in seconds and deliver a structured, validated routing decision — ready for the specialist team to act on.

This same prompt-chaining + gate-check pattern can be applied to any domain where you need to classify, assess, and route unstructured text input: medical triage, legal document sorting, customer support tickets, and more.
